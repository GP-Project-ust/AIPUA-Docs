## 1. المقدمة

### 1.1 الغرض من الوثيقة

تحدد هذه الوثيقة رؤية عامة وشاملة لمشروع «المساعد الذكي المؤسسي»، وتصف الدوافع والأهداف والنطاق وأصحاب المصلحة والميزات الرئيسية والمعمارية على مستوى الرؤية. تهدف الوثيقة إلى توفير فهم مشترك بين فريق التطوير والمشرف الأكاديمي ولجنة المناقشة وأي طرف ذي علاقة بطبيعة المشروع ومساره. كما تمثل هذه الوثيقة المرجع الأساسي للقرارات التصميمية والتنفيذية اللاحقة، وتسهم في توجيه عملية بناء النظام بطريقة منهجية ومتسقة مع الأهداف المسطرة.

### 1.2 نطاق الوثيقة

تشمل هذه الوثيقة وصفاً عالياً للمشكلة التي يعالجها النظام، والحل المقترح، والفئة المستهدفة، والمكونات الأساسية، والمخرجات المتوقعة، والقيود، والافتراضات. كما تتضمن وصفاً لمراحل النظام الرئيسية (التكامل الذكي والتشغيل)، والتقنيات المقترحة، وحالات الاستخدام النموذجية. لا تتطرق الوثيقة إلى التفاصيل التقنية العميقة (كتصميم قاعدة البيانات أو مواصفات الـ APIs)، إذ تُترك هذه التفاصيل لوثيقة متطلبات البرمجيات (SRS) ووثيقة تصميم النظام (SDD).

### 1.3 التعريفات والاختصارات

| المصطلح | التعريف |
|---------|---------|
| LLM | نماذج اللغة الكبيرة (Large Language Models) |
| RAG | التوليد المعزز بالاسترجاع (Retrieval-Augmented Generation) |
| Text-to-SQL | تحويل النص الطبيعي إلى استعلام SQL |
| RBAC | التحكم بالوصول القائم على الأدوار |
| Single-Tenant | معمارية يخدم فيها كل deployment مؤسسة واحدة |
| Prototype | نموذج أولي قابل للتشغيل والتخصيص |
| Smart Onboarding | مرحلة تكامل ذكية لتخصيص النظام لمؤسسة جديدة |
| Institution Profile | ملف تعريف المؤسسة (YAML) يحتوي على التخصيصات الدلالية |
| Multi-Agent | معمارية الوكلاء المتعددين (Agentic AI) |
| LangGraph | إطار عمل لتنسيق الوكلاء الأذكياء |
| Database Views Layer | طبقة المشاهد المبسطة لتقليل تعقيد SQL |
| Single-LLM Architecture | معمارية تعتمد على نموذج لغوي واحد قابل للتبديل |
| Schema Introspection | اكتشاف تلقائي لبنية قاعدة البيانات |
| Ollama | أداة مفتوحة المصدر لتشغيل نماذج اللغة محلياً |
| Vector DB | قاعدة بيانات شعاعية لتخزين التضمينات النصية |
| MVP | الحد الأدنى من المنتج القابل للتشغيل |

### 1.4 المراجع

- وثيقة متطلبات البرمجيات (SRS) — مرجعية رئيسية
- وثيقة تصميم النظام (SDD) — تُكتب لاحقاً
- قائمة المشاريع المعلنة من القسم — المشروع رقم 27
- توثيقات Ollama, LangChain, LangGraph, FastAPI, Next.js الرسمية

### 1.5 نظرة عامة على الوثيقة

يتكون هذا المستند من تسعة أقسام رئيسية: المقدمة، وضعية المشروع، أصحاب المصلحة، نظرة عامة على المنتج، الميزات الرئيسية، المعمارية على مستوى الرؤية، القيود، مراحل النظام والتقنيات وحالات الاستخدام، والخلاصة.

## 2. وضعية المشروع

### 2.1 فرصة العمل / المشكلة

تعاني الكثير من المؤسسات التعليمية والإدارية من تشتت بياناتها عبر أنظمة متعددة وغير متكاملة، مما يجعل الوصول إلى المعلومات وتحليلها مهمة معقدة تتطلب خبراء تقنيين. كما تواجه هذه المؤسسات تحدياً في توفير خدمات استعلام سريعة ودقيقة للمستخدمين، حيث تتطلب الاستعلامات الروتينية وقتاً طويلاً وجهداً بشرياً كبيراً، مما يقلل من كفاءة العمل ويؤثر على جودة الخدمة المقدمة.

من ناحية أخرى، أحدثت نماذج اللغة الكبيرة ثورة في القدرة على فهم اللغة الطبيعية، لكن استخدامها مباشرة في البيانات المؤسسية يواجه تحديات أمنية ودقة وتكامل. لا يوجد حالياً حل واحد يجمع بين قدرات الـ LLMs والربط المرن بقواعد البيانات المؤسسية مع ضمان الأمان والصلاحيات — وهي الفجوة التي يسدها هذا المشروع.

### 2.2 بيان المشكلة

تعاني المؤسسات من صعوبة في الاستفادة من بياناتها الموزعة عبر أنظمة متعددة، وغياب واجهة استعلام موحدة تسمح للمستخدمين غير التقنيين بالوصول إلى المعلومات بلغة طبيعية. يؤثر ذلك على سرعة القرارات وجودة الخدمة المقدمة للجمهور. إذا تمكن فريق المشروع من بناء مساعد ذكي مؤسسي يعتمد على الذكاء الاصطناعي ويتصل بقواعد بيانات المؤسسة بطريقة آمنة ومرنة، فإن ذلك سيقلل من الوقت اللازم للاستعلامات ويحسّن جودة القرارات. على العكس، إذا لم يُعالج هذا التحدي، فستستمر المؤسسات في الاعتماد على الطرق اليدوية البطيئة.

### 2.3 بيان الرؤية

> 💡 **«تمكين أي مؤسسة من تحويل بياناتها الموزعة إلى معرفة قابلة للاستعلام، عبر مساعد ذكي يتحدث العربية، يحترم صلاحيات المستخدمين، ويحافظ على خصوصية بياناتهم، دون الحاجة إلى خبراء تقنيين.»**

تتمحور هذه الرؤية حول ثلاث قيم أساسية: **الذكاء** (القدرة على فهم اللغة الطبيعية وتقديم إجابات دقيقة)، و**الأمان** (ضمان الخصوصية والصلاحيات الصارمة)، و**الشمولية** (قابلية التطبيق في أي مؤسسة دون تعديل برمجي).

### 2.4 بيان المهمة

بناء **نموذج أولي (Prototype)** لمساعد ذكي مؤسسي يعتمد على **نموذج لغوي واحد قابل للتبديل (Single-LLM Architecture)** — محلياً عبر Ollama أو عبر API سحابي — يُدار بالكامل من لوحة التحكم. يعتمد النظام على **معمارية الوكلاء المتعددين (Multi-Agent Architecture)** لدمج تقنيات RAG و Text-to-SQL و LLM Reasoning، ويربط بأي قاعدة بيانات عبر **طبقة مشاهد مبسطة (Database Views Layer)** وواجهة مرنة دون تعديل الكود، مع نظام صلاحيات قائم على الأدوار ومرحلة تكامل ذكية (Smart Onboarding)، خلال مدة لا تتجاوز ثمانية أشهر، بحيث يكون **قابلاً للدمج والتخصيص لأي مؤسسة واحدة**.

## 4. نظرة عامة على المنتج

### 4.1 نظرة عامة على المنتج

المساعد الذكي المؤسسي هو **نموذج أولي (Prototype)** لنظام برمجي متعدد الطبقات صُمّم ليكون **قابلاً للتخصيص والدمج في أي مؤسسة تعليمية أو إدارية واحدة**. يعتمد النظام على فصل واضح للمسؤوليات (Separation of Concerns) بين الطبقات، حيث تتولى كل طبقة وظيفة محددة وتتواصل مع الطبقات الأخرى عبر واجهات برمجية موثقة (API Contracts). هذا التصميم يضمن قابلية الصيانة والتوسعة، ويسمح باستبدال أي مكون دون التأثير على باقي النظام.

يُنشر النظام بمعمارية **Single-Tenant** — أي أن كل deployment يخدم مؤسسة واحدة فقط، مما يبسّط التصميم ويرفع الأداء ويفصل التعقيدات الأمنية المرتبطة بالعزل بين المؤسسات. هذا القرار يجعل النظام أكثر واقعية كمشروع تخرج وأسرع في التطوير، مع بقاء الباب مفتوحاً للتوسعة إلى Multi-Tenant في المستقبل.

### 4.2 افتراضات المنتج

- توفر الأجهزة اللازمة لتشغيل النموذج المحلي عبر Ollama (GPU بـ 16GB VRAM على الأقل)، أو توفر مفتاح API سحابي.
- توفر اتصال إنترنت مستقر لتحميل النموذج والحزم البرمجية أثناء التطوير، وللوصول إلى الـ APIs السحابية عند اختيارها.
- التزام أعضاء الفريق بالجدول الزمني المحدد.
- توفر الإشراف الأكاديمي المستمر من قبل المشرف.
- قبول البيانات التجريبية المولّدة كمحاكاة كافية لبيئة مؤسسية حقيقية لأغراض التقييم الأكاديمي.

### 4.3 الافتراضات والقيود

**الافتراضات:**
- نموذج Qwen 2.5 سيستمر في دعم اللغة العربية بشكل جيد في الإصدارات المستقبلية.
- إطار العمل LangChain و LangGraph سيبقيان مستقرين ومتوافقين مع التحديثات المستقبلية.
- البيانات التجريبية المولّدة ستمثل بدقة كافية بيئة مؤسسية حقيقية لأغراض التقييم الأكاديمي.
- توفر اتصال إنترنت موثوق عند اختيار النموذج السحابي عبر API.

**القيود الأساسية:**
- مدة المشروع: 8 أشهر (فصلان دراسيان).
- حجم الفريق: 4 أعضاء فقط.
- الترخيص: استخدام تقنيات مفتوحة المصدر فقط.
- البيانات: استخدام بيانات تجريبية مولّدة بدلاً من بيانات حقيقية لأسباب الخصوصية.
- النشر: على خادم محلي واحد (Single-Node Deployment) لكل مؤسسة.
- المعمارية: Single-Tenant (deployment واحد = مؤسسة واحدة).
- النموذج اللغوي: نموذج واحد فقط (محلي أو سحابي) يُحدد عبر لوحة التحكم.

## 5. الميزات الرئيسية

### 5.1 🎯 التفاعل الذكي بلغة طبيعية

النظام يفهم الاستفسارات باللغة العربية (فصحى وعامية) ويجيب عليها بطريقة طبيعية ومفهومة. يدعم النظام ثلاثة أنواع من الأسئلة: (1) أسئلة إحصائية عن البيانات المنظمة (Text-to-SQL)، (2) أسئلة معرفية عن الوثائق واللوائح (RAG)، (3) أسئلة تحليلية استنتاجية (LLM Reasoning). كما يعرض النظام المصادر المستخدمة في كل إجابة لضمان الشفافية وإمكانية التحقق، ويقترح أسئلة ذات صلة لتشجيع الاستكشاف.

### 5.2 🔌 التكامل المرن مع أي قاعدة بيانات

النظام يرتبط بأي قاعدة بيانات (PostgreSQL, MySQL, MongoDB, SQL Server, Oracle) عبر Adapters قابلة للتبديل دون تعديل الكود. يتم اكتشاف بنية الجداول تلقائياً (Schema Introspection)، ويمكن لصاحب النظام إضافة شروحات نصية (Annotations) لتوضيح المعاني والقواعد التنظيمية. يتم تطبيق فلترة على مستوى الصفوف (Row-Level Security) حسب صلاحيات المستخدم.

### 5.3 🔐 نظام صلاحيات مرن وقائم على الأدوار

يوفّر النظام إطاراً مرناً يسمح لصاحب النظام بتعريف أدوار مخصصة (مثل: طالب، أستاذ، إداري، مدير، مسؤول نظام) وصلاحياتها دون كتابة كود. يمكن تحديد الصلاحيات على مستوى: مؤسسة، قسم، سجل، أو حقل. تُطبّق حدود معدل الاستعلامات (Rate Limiting) حسب الدور، وتُسجّل كل العمليات في Audit Log للمراجعة الأمنية.

### 5.4 🤖 معمارية الوكلاء المتعددين (Multi-Agent Architecture) ⭐

يعتمد النظام على **معمارية الوكلاء المتعددين (Agentic Workflow)** بدلاً من التوجيه البسيط، باستخدام إطار العمل **LangGraph**. تنقسم مهام الذكاء الاصطناعي بين وكلاء متخصصين:

- **Supervisor Agent (الوكيل المشرف)**: يحلل السؤال، يقرر أي الوكلاء يحتاجهم، يجمع إجاباتهم، ويقدم الإجابة النهائية.
- **SQL Agent (وكيل قواعد البيانات)**: متخصص في تحويل الأسئلة إلى SQL عبر طبقة المشاهد المبسطة.
- **RAG Agent (وكيل الوثائق)**: متخصص في البحث في الوثائق واللوائح.
- **Analyzer Agent (وكيل التحليل)**: يجمع نتائج الوكلاء الآخرين ويقدم تحليلات استنتاجية وتوصيات.

**الفائدة**: فصل المسؤوليات يرفع الدقة، يقلل الهلوسة، ويسمح بتطوير كل وكيل بشكل مستقل. هذا التوجه يمثل أحدث ما في علم الـ Agentic AI ويعطي ثقلاً علمياً كبيراً للمشروع.

### 5.5 🗄️ طبقة المشاهد المبسطة (Database Views Layer) ⭐

بدلاً من جعل الـ LLM يقرأ جداول النظام المعقدة مباشرة (والتي قد تحتوي على مئات الأعمدة غير المفهومة وعلاقات متشابكة)، يضيف النظام **طبقة وسيطة من المشاهد (Views)** تُنشئ مشاهد مبسطة ومجمّعة تلقائياً أثناء مرحلة التكامل.

- **المشكلة**: استعلام `JOIN` معقد بين 5 جداول يربك الـ LLM ويخفض الدقة إلى 50%.
- **الحل**: إنشاء `V_Active_Students` كمشهد جاهز يجمع البيانات، فيرى الـ LLM جدولاً واحداً بسيطاً.
- **الفائدة**: ترفع دقة الـ Text-to-SQL بشكل هائل (من 70% إلى 90%+) وتمنع أخطاء العلاقات المتشابكة.

### 5.6 ⚡ مرحلة التكامل الذكي (Smart Onboarding)

صُمّم النظام كنموذج أولي **قابل للدمج والتخصيص لأي مؤسسة واحدة** بسرعة عبر معالج تكامل ذكي شبه مؤتمت. يستخدم الذكاء الاصطناعي لتحليل بنية البيانات وبناء «فهم دلالي» للنظام، مما يحول عملية تخصيص النظام لمؤسسة جديدة من أسابيع إلى ساعات. تتضمن مرحلة التكامل توليد Institution Profile (YAML)، بناء Database Views تلقائياً، اختبار الأسئلة القياسية، وتقرير جاهزية للإطلاق.

### 5.7 🧠 نموذج لغوي واحد قابل للتبديل (Single-LLM Architecture) ⭐

يعتمد النظام على **نموذج لغوي واحد فقط** في كل deployment، يمكن اختياره بين:
- **نموذج محلي** عبر Ollama (مثل Qwen 2.5) — لخصوصية كاملة وتشغيل بدون إنترنت.
- **نموذج سحابي** عبر API (مثل OpenAI GPT-4o، Claude، DeepSeek) — لأداء أعلى بدون متطلبات أجهزة.

**التحكم عبر Dashboard**: يمكن للمدير التبديل بين المحلي والسحابي، أو تغيير النموذج، أو ضبط المعاملات (Temperature, Max Tokens) — كل ذلك دون تعديل الكود. هذا التصميم يبسّط البنية التحتية، يقلل التكلفة، ويسمح لكل مؤسسة باختيار ما يناسبها.

**ملاحظة معمارية**: جميع الوكلاء (Supervisor, SQL, RAG, Analyzer) يستخدمون نفس النموذج المختار، مع System Prompts مختلفة لكل وكيل تحدد دوره. هذا يضمن اتساق الإجابات ويبسط الإدارة.

### 5.8 📊 توليد التقارير المخصصة

يولّد النظام تقارير رسمية بقوالب قابلة للتخصيص لكل مؤسسة، مع إضافة شعار المؤسسة وتنسيقها الرسمي. تُصدَّر التقارير بصيغ PDF, Excel, Word. يتم التوليد بشكل غير متزامن (Asynchronous) عبر Celery Worker لتجنب انقطاع الاتصال، ويُرسل إشعار للمستخدم عند الاكتمال.

### 5.9 🛡️ أمان متعدد الطبقات

يضع النظام الأمان في الصدارة عبر أربع طبقات حماية للجداول:
- 🔴 **Blacklist**: جداول محذوفة كلياً (مثل passwords, audit_keys).
- 🟠 **Restricted**: صلاحية خاصة (مثل salaries, bank_accounts).
- 🟡 **Masked**: إخفاء جزئي للأعمدة الحساسة (مثل الرقم القومي).
- 🟢 **Public**: متاحة لكل الأدوار حسب RBAC.

مع إضافة: مستخدم DB للقراءة فقط، SQL Validator، تشفير AES-256، TLS 1.3، Audit Logs شامل.

### 5.10 🌐 دعم اللغة العربية الكامل

النظام يدعم العربية في كل المستويات: واجهة المستخدم (RTL)، System Prompts، رسائل الأخطاء، التقارير المولّدة، الاستعلامات. يدعم الفصحى والعامية، وتنسيق التواريخ بالتقويمين الهجري والميلادي.

#### الوكلاء المتخصصون:

| الوكيل | المسؤولية | System Prompt مخصص |
|--------|----------|-------------------|
| **Supervisor Agent** | تحليل السؤال، توزيع المهام، جمع الإجابات، التنسيق | "أنت مشرف ذكي، حلل السؤال وقرر..." |
| **SQL Agent** | تحويل الأسئلة لـ SQL عبر Database Views، تنفيذها | "أنت خبير SQL، استخدم Views فقط..." |
| **RAG Agent** | البحث في الوثائق، توليد الإجابات المعرفية | "أنت باحث وثائق، استخدم المصادر..." |
| **Analyzer Agent** | جمع نتائج الوكلاء، التحليل الاستنتاجي، التوصيات | "أنت محلل بيانات، قدم توصيات..." |
| **Report Agent** | توليد التقارير الرسمية بناءً على نتائج الوكلاء | "أنت كاتب تقارير رسمية..." |

#### ملاحظة معمارية مهمة:
جميع الوكلاء يستخدمون **نفس النموذج اللغوي الواحد** المختار من الـ Dashboard (محلي أو سحابي)، مع System Prompts مختلفة لتحديد دور كل وكيل. هذا التصميم يحقق:
- ✅ بساطة الإدارة (نموذج واحد)
- ✅ اتساق الإجابات
- ✅ تقليل التكلفة
- ✅ مرونة التبديل

### 6.3 طبقة المشاهد المبسطة (Database Views Layer) ⭐

#### المشكلة التي تحلها:

عند توصيل LLM بـ 1000 جدول بعلاقات متشابكة، يواجه مشكلتين:
1. **ضياع في الفهم**: مئات الأعمدة غير المفهومة
2. **أخطاء الـ JOINs**: استعلامات معقدة تخفض الدقة إلى 50%

#### الحل المعماري:

بدلاً من قراءة الجداول الأصلية مباشرة، يتعامل SQL Agent مع **طبقة مشاهد مبسطة (Views)** تُنشأ تلقائياً أثناء Smart Onboarding:

```sql
-- بدلاً من جعل LLM يكتب:
SELECT s.name, c.college_name, g.gpa
FROM STD_INFO s
JOIN COLLEGES c ON s.college_cd = c.college_cd
JOIN STD_GRADES g ON s.std_no = g.std_no
WHERE s.std_stat = '01' AND g.sem_code = '20251';

-- يقوم LLM بكتابة:
SELECT name, college_name, gpa
FROM V_Active_Students_GPA
WHERE semester = '20251';

-- ← أبسط، أوضح، أدق!
```

#### كيف تُنشأ المشاهد؟

أثناء Smart Onboarding:
1. النظام يحلل الجداول والعلاقات
2. يحدد الأنماط الشائعة (Active Students, Recent Grades, etc.)
3. يولّد SQL لإنشاء Views مبسطة
4. يسجّلها في Institution Profile
5. يعرضها لـ SQL Agent بدلاً من الجداول الأصلية

#### الفوائد:
- 📈 **رفع دقة SQL من 70% إلى 90%+**
- 🛡️ تقليل أخطاء الـ JOINs المتشابكة
- ⚡ استعلامات أسرع (Views محسّنة)
- 🧹 تبسيط مهمة LLM
- 🔒 طبقة أمان إضافية (Views لا تكشف الجداول الأصلية)

### 6.4 طبقة النموذج اللغوي الموحدة (Single-LLM Layer) ⭐

#### المبدأ:

بدلاً من استخدام نماذج متعددة لكل مهمة، يعتمد النظام على **نموذج واحد فقط** يخدم جميع الوكلاء، مع تجريد كامل عبر LLM Abstraction Layer.

#### الخيارات المتاحة:

| النوع | المزود | النماذج المدعومة | الميزة |
|------|--------|-----------------|--------|
| **محلي** | Ollama | Qwen 2.5 (7B/14B), Llama 3.1, Mistral | خصوصية كاملة، لا إنترنت، تشغيل مجاني |
| **سحابي** | OpenAI | GPT-4o, GPT-4o-mini | أعلى دقة، لا يحتاج GPU |
| **سحابي** | Anthropic | Claude 3.5 Sonnet | ممتاز للتحليل، سياق ضخم |
| **سحابي** | DeepSeek | DeepSeek V3, DeepSeek R1 | رخيص، أداء جيد |

#### التحكم عبر Dashboard:

```yaml
# إعداد النموذج من لوحة التحكم
llm_config:
  mode: "local"           # "local" OR "cloud"
  
  # عند اختيار المحلي:
  local:
    provider: "ollama"
    model: "qwen2.5:7b"
    endpoint: "http://localhost:11434"
    temperature: 0.3
    max_tokens: 4096
    timeout: 60
  
  # عند اختيار السحابي:
  cloud:
    provider: "openai"    # "openai" | "anthropic" | "deepseek"
    model: "gpt-4o"
    api_key: "********"
    temperature: 0.3
    max_tokens: 4096
    timeout: 30
```

#### الفوائد:
- ✅ **بساطة الإدارة**: نموذج واحد لإدارته
- ✅ **اتساق الإجابات**: جميع الوكلاء يستخدمون نفس العقل
- ✅ **تقليل التكلفة**: لا حاجة لتشغيل نماذج متعددة
- ✅ **مرونة التبديل**: التبديل بين المحلي والسحابي بضغطة زر
- ✅ **ملاءمة للمؤسسة**: كل مؤسسة تختار ما يناسبها

### 6.5 نهج الذكاء الاصطناعي الهجين (Hybrid AI)

النظام يدرك أن نوعاً واحداً من المعالجة لا يكفي للإجابة على جميع أنواع الاستفسارات المؤسسية:

| نوع الاستفسار | الوكيل المسؤول | مثال |
|--------------|---------------|------|
| إحصائي | SQL Agent | «كم عدد الطلاب المنسحبين؟» |
| معرفي | RAG Agent | «ما لوائح التخرج؟» |
| تحليلي | Analyzer Agent | «حلل أسباب الانسحاب» |
| مركب | Multi-Agent (الكل) | «كم عدد المنسحبين وحلل الأسباب» |

### 6.6 Smart Onboarding (مرحلة التكامل الذكي)

معالج شبه مؤتمت لتخصيص النظام لمؤسسة جديدة، ينتج:
1. Institution Profile (YAML) — القلب الدلالي
2. Database Views Layer — المشاهد المبسطة ⭐
3. Validated SQL Examples (JSON) — Few-shot Training
4. Canonical Schema Map — خريطة العلاقات

6. Onboarding Checklist — قائمة التحقق

## 8. مراحل النظام: التكامل والتشغيل والتقنيات وحالات الاستخدام

> ⭐ **هذا القسم يجمع المراحل الرئيسية للنظام، التقنيات المقترحة، وحالات الاستخدام النموذجية**

### 8.1 المرحلة الأولى: التكامل الذكي (Smart Onboarding)

#### 8.1.1 نظرة عامة

مرحلة التكامل الذكي هي عملية **شبه مؤتمتة** لتخصيص النظام لمؤسسة جديدة. تستخدم الذكاء الاصطناعي لتحليل بنية البيانات وبناء «فهم دلالي» للنظام. الهدف: **تخصيص نظام لمؤسسة بـ 1000 جدول خلال 8 ساعات بدلاً من أسابيع**.

#### 8.1.2 المخرجات الخمسة للمرحلة

```
1️⃣ Institution Profile (YAML)        ← القلب الدلالي
2️⃣ Database Views Layer (SQL Views)  ← المشاهد المبسطة ⭐
3️⃣ Validated SQL Examples (JSON)     ← Few-shot Training
4️⃣ Canonical Schema Map (Visual)     ← الفهرس الفعلي
5️⃣ Validation Report (PDF)           ← تقرير الجاهزية
```

#### 8.1.3 خطوات مرحلة التكامل (8 ساعات)

**الخطوة 1: الاتصال والاكتشاف (ساعة 1-2)**
- المدير يدخل بيانات الاتصال بقاعدة البيانات
- النظام يكتشف الجداول تلقائياً (Schema Introspection)
- المدير يحدد الجداول المهمة
- النظام يصنّف الجداول تلقائياً (طلاب، مالية، موارد بشرية...)

**الخطوة 2: الذكاء يشتغل (ساعة 3-4)**
- النموذج اللغوي يحلل كل جدول (الاسم، الأعمدة، عينة بيانات)
- يستنتج المعاني (STD_INFO = جدول الطلاب)
- يقترح Canonical Mapping تلقائياً
- يولّد Institution Profile أولي
- **يولّد Database Views المبسطة تلقائياً** ⭐
- يولّد 30 سؤالاً تجريبياً + SQL متوقع
- يختبرها على DB فعلي

**الخطوة 3: المراجعة والتصحيح (ساعة 5-6)**
- المدير يفتح «وضع المراجعة» في Dashboard
- يرى الجداول المكتشفة + Mapping المقترح
- يرى Views المولّدة ويمكن تعديلها
- يصحّح أي خطأ بنقرات
- يوافق على الأسئلة الصحيحة
- يضبط الأمان (Blacklist/Restricted/Masked/Public)
- يضيف أسئلة خاصة بالمؤسسة

**الخطوة 4: الإطلاق (ساعة 7-8)**
- Dashboard يعرض: «دقة SQL: 90% — جاهز للإطلاق»
- المدير يضغط «تفعيل المؤسسة»
- النظام جاهز للاستخدام

#### 8.1.4 المزامنة المستمرة

بعد الإطلاق، يستمر النظام في مزامنة الـ Schema:
- **مزامنة فورية**: تنبيه عند إضافة/حذف جدول
- **مزامنة دورية**: إعادة فحص الـ Schema ليلاً
- **مزامنة بطلب**: زر «إعادة المزامنة» في الـ Dashboard

### 8.2 المرحلة الثانية: التشغيل (Operation)

#### 8.2.1 نظرة عامة

بعد اكتمال Smart Onboarding، يدخل النظام مرحلة التشغيل المنتظم، حيث يستخدم المستخدمون النظام عبر واجهة الدردشة، ويقوم مسؤول النظام بالإدارة والمراقبة.

#### 8.2.2 دورة حياة الاستفسار

```
1️⃣ المستخدم يكتب سؤالاً بالعربية
        │
        ▼
2️⃣ بوابة الـ API تتأكد من المصادقة والصلاحيات
        │
        ▼
3️⃣ Supervisor Agent يحلل السؤال (باستخدام النموذج المختار)
        │
        ▼
4️⃣ Supervisor يوزع المهام على الوكلاء:
   ├─ SQL Agent → يحول السؤال لـ SQL عبر Views
   ├─ RAG Agent → يبحث في الوثائق
   └─ Analyzer Agent → (في حالة الأسئلة المركبة)
        │
        ▼
5️⃣ SQL Agent ينفّذ SQL على DB (بعد Validation + RLS)
        │
        ▼
6️⃣ Supervisor يجمع النتائج
        │
        ▼
7️⃣ Analyzer Agent (اختياري) يحلل ويقدم توصيات
        │
        ▼
8️⃣ Supervisor يركّب الإجابة النهائية بالعربية
        │
        ▼
9️⃣ النظام يعرض الإجابة + المصادر + SQL المولّد
        │
        ▼
🔟 تسجيل العملية في Audit Log
```

#### 8.2.3 وحدات التشغيل الرئيسية

| الوحدة | الوظيفة |
|--------|---------|
| وحدة المصادقة | تسجيل الدخول، JWT، 2FA |
| وحدة الدردشة | استقبال الأسئلة، عرض الإجابات Streaming |
| وحدة Multi-Agent | تنسيق الوكلاء وتنفيذ الأسئلة |
| وحدة LLM | إدارة النموذج المختار (محلي/سحابي) |
| وحدة التقارير | توليد التقارير بشكل غير متزامن |
| وحدة الإدارة | إدارة المستخدمين، الأدوار، النماذج، LLM Config |
| وحدة المراقبة | Health Dashboard، Audit Logs، إحصائيات |
| وحدة الأمان | SQL Validator، RLS، Rate Limiting |

#### 8.2.4 أنواع الاستفسارات المدعومة

| النوع | الوكيل المسؤول | مثال |
|------|---------------|------|
| إحصائي بسيط | SQL Agent | «كم عدد الطلاب في كلية الهندسة؟» |

### 8.4 حالات الاستخدام (Use Cases)

#### UC-001: استعلام إحصائي عبر الدردشة

- **الفاعل**: مستخدم لديه صلاحية استعلام
- **الشرط المسبق**: مسجّل للدخول، لديه صلاحية الاستعلام
- **التدفق الأساسي**:
  1. يفتح المستخدم واجهة الدردشة
  2. يكتب استفساراً إحصائياً (مثلاً: «كم عدد الطلاب المنسحبين من قسم التمريض بين 2011 و2014؟»)
  3. Supervisor Agent يحلل السؤال ويوجّهه لـ SQL Agent
  4. SQL Agent يحوله لاستعلام SQL عبر Database Views
  5. SQL Validator يتحقق من السلامة
  6. النظام يتحقق من صلاحيات المستخدم (RBAC + RLS)
  7. SQL Agent ينفّذ الاستعلام على قاعدة البيانات
  8. Supervisor Agent يحول النتيجة إلى نص عربي
  9. النظام يعرض الإجابة مع SQL المولّد للمستخدم
- **التدفقات البديلة**: إذا كان المستخدم يفتقر للصلاحية، يعرض رسالة مناسبة. إذا فشل توليد SQL، يقترح إعادة صياغة السؤال.
- **الشروط اللاحقة**: تسجيل عملية الاستعلام في Audit Log، حفظ المحادثة.

#### UC-002: استفسار عن لوائح أو وثائق

- **الفاعل**: أي مستخدم مسجّل للدخول
- **الشرط المسبق**: مسجّل للدخول، الوثائق مرفوعة في النظام
- **التدفق الأساسي**:
  1. يكتب المستخدم سؤالاً معرفياً (مثلاً: «ما هي متطلبات التخرج من تخصص X؟»)
  2. Supervisor Agent يحلل السؤال ويوجّهه لـ RAG Agent
  3. RAG Agent يولّد تضمين للسؤال (Embedding)
  4. RAG Agent يبحث في Qdrant عن المقاطع الأكثر صلة
  5. RAG Agent يمرر المقاطع مع السؤال لنموذج اللغة لتوليد الإجابة
  6. Supervisor Agent يعرض الإجابة مع المصادر (اسم الوثيقة، رقم الصفحة)
- **التدفقات البديلة**: إذا لم يجد مقاطع ذات صلة، يعرض رسالة «لا توجد معلومات كافية».

#### UC-003: توليد تقرير شامل

- **الفاعل**: مستخدم لديه صلاحية توليد تقارير
- **الشرط المسبق**: مسجّل للدخول، لديه صلاحية
- **التدفق الأساسي**:
  1. يكتب المستخدم في الدردشة: «ولّد تقريراً عن أداء قسم X في الفصل الأول 2024»
  2. Supervisor Agent يحلل الطلب ويوجّهه لـ Report Agent
  3. Report Agent يطلب من SQL Agent جمع البيانات اللازمة
  4. Report Agent يختار القالب المناسب للتقرير
  5. النظام يرسل مهمة توليد التقرير إلى Celery Worker
  6. النظام يخطر المستخدم بأن التقرير قيد التوليد
  7. العامل يكمل توليد التقرير خلال 60 ثانية
  8. النظام يرسل إشعاراً للمستخدم باكتمال التقرير
  9. المستخدم يحمل التقرير بصيغة PDF
- **التدفقات البديلة**: إذا فشل توليد التقرير، يعرض رسالة خطأ ويسمح بإعادة المحاولة.

#### UC-004: Smart Onboarding لمؤسسة جديدة

- **الفاعل**: مسؤول النظام
- **الشرط المسبق**: مسجّل للدخول بصلاحية Super Admin، يملك بيانات اعتماد DB
- **التدفق الأساسي**:
  1. يفتح المسؤول Dashboard → «إضافة مؤسسة جديدة»
  2. يختار نوع DB + يدخل بيانات الاتصال
  3. النظام يتصل → يكتشف 1000+ جدول تلقائياً
  4. النموذج اللغوي يصنّف الجداول إلى مجموعات (طلاب، مالية، إلخ)
  5. المسؤول يحدد الجداول المهمة + يحظر الحساسة
  6. النموذج يحلل كل جدول + يقترح Mapping تلقائياً
  7. النظام يولّد Institution Profile أولي
  8. **النظام يولّد Database Views المبسطة تلقائياً** ⭐
  9. النظام يولّد 30 سؤالاً تجريبياً + SQL متوقع
  10. النظام يختبرها على DB فعلي → يحسب الدقة
  11. المسؤول يفتح «وضع المراجعة» → يصحّح الأخطاء
  12. النظام يعيد التقييم بعد التصحيح
  13. Dashboard يعرض: «دقة SQL: 90% — جاهز للإطلاق»
  14. المسؤول يضغط «تفعيل المؤسسة»
- **التدفقات البديلة**: إذا فشل الاتصال، يعرض رسالة خطأ تفصيلية.
- **الشروط اللاحقة**: المؤسسة مفعّلة، Profile + Views محفوظة، تسجيل العملية في Audit Log.

#### UC-005: تحليل أسباب ظاهرة معينة (سؤال مركب)

- **الفاعل**: مستخدم لديه صلاحية تحليل
- **الشرط المسبق**: مسجّل للدخول، يملك صلاحية وصول للبيانات
- **التدفق الأساسي**:
  1. يكتب المستخدم: «حلل أسباب انسحاب الطلاب من قسم التمريض بين 2011-2014»
  2. Supervisor Agent يحلل السؤال ويصنفه كسؤال مركب
  3. Supervisor يوزع المهام:
     - SQL Agent: إحصاء أعداد المنسحبين حسب السبب
     - RAG Agent: البحث عن تقارير الانسحاب وملاحظات المرشدين
  4. SQL Agent ينفّذ استعلاماً عبر Views
  5. RAG Agent يجلب المقاطع ذات الصلة
  6. Analyzer Agent يجمع النتائج الإحصائية والتحليلات النصية
  7. Analyzer Agent يولّد تحليلاً متماسكاً مع توصيات
  8. Supervisor Agent يعرض التحليل مع رسوم بيانية ومصادر
  9. Supervisor يقترح توليد تقرير رسمي بالنتائج
- **التدفقات البديلة**: إذا كانت بيانات الأسباب غير متوفرة، يحلل النموذج العوامل المرتبطة فقط.

#### UC-006: تبديل النموذج اللغوي (محلي ↔ سحابي) ⭐

- **الفاعل**: مسؤول النظام
- **الشرط المسبق**: مسجّل للدخول بصلاحية System Admin
- **التدفق الأساسي**:
  1. يفتح المسؤول Dashboard → «إدارة النموذج اللغوي»
  2. يرى الإعداد الحالي (مثلاً: محلي - Qwen 2.5 7B)
  3. يختار التبديل إلى: سحابي - OpenAI GPT-4o
  4. يدخل API Key ويضبط المعاملات
  5. يضغط «اختبار الاتصال»
  6. النموذج الجديد يستجيب لسؤال تجريبي
  7. المسؤول يضغط «حفظ وتفعيل»
  8. النظام يحوّل جميع الوكلاء لاستخدام النموذج الجديد فوراً
  9. النظام يعرض رسالة تأكيد
- **التدفقات البديلة**: إذا فشل اختبار الاتصال، يعرض رسالة خطأ ولا يسمح بالحفظ.
- **الشروط اللاحقة**: جميع الوكلاء يستخدمون النموذج الجديد، تسجيل العملية في Audit Log.

#### UC-007: إضافة نموذج LLM جديد

- **الفاعل**: مسؤول النظام
- **الشرط المسبق**: مسجّل للدخول، النموذج مثبت في Ollama أو يتوفر API key
- **التدفق الأساسي**:
  1. يفتح المسؤول قسم «إدارة النماذج»
  2. ينقر على «إضافة نموذج جديد»
  3. يختار نوع النموذج (محلي/سحابي)
  4. يملأ تفاصيل النموذج (المزود، الاسم، الـ endpoint، API key)
  5. يضبط المعاملات (Temperature, Max Tokens, Timeout)
  6. ينقر على «اختبار النموذج» بإدخال سؤال تجريبي
  7. النظام يعرض استجابة النموذج للسؤال التجريبي
  8. بعد التحقق من جودة الاستجابة، ينقر على «حفظ»
  9. النموذج الجديد يصبح متاحاً للتفعيل من الـ Dashboard
- **التدفقات البديلة**: إذا فشل اختبار النموذج، يعرض رسالة خطأ ولا يسمح بالحفظ.

#### UC-008: تعريف دور جديد وصلاحياته

- **الفاعل**: مسؤول النظام
- **الشرط المسبق**: مسجّل للدخول بصلاحية إدارة الأدوار
- **التدفق الأساسي**:
  1. يفتح المسؤول قسم «إدارة الأدوار»
  2. ينقر على «إنشاء دور جديد»
  3. يحدد اسم الدور ووصفه
  4. يحدد الصلاحيات على كل مورد (Read/Write/Admin)
  5. يحدد نطاق الصلاحيات (مؤسسة/قسم/سجل/حقل)
  6. ينقر على «حفظ»
  7. يصبح الدور متاحاً لتخصيصه للمستخدمين
- **التدفقات البديلة**: إذا كان اسم الدور مستخدماً، يحذر النظام المسؤول.

#### UC-009: المزامنة التلقائية عند تغيير DB

- **الفاعل**: النظام (تلقائي)
- **الشرط المسبق**: النظام يعمل، DB المؤسسة قد تغيّرت (جدول جديد/محذوف)
- **التدفق الأساسي**:
  1. النظام يكتشف تغييراً في schema قاعدة البيانات
  2. النظام يرسل تنبيهاً فورياً للمدير: «تم اكتشاف جدول جديد: STUDENT_ATTENDANCE_2025»
  3. النموذج اللغوي يحلل الجدول الجديد تلقائياً
  4. النظام يقترح Mapping + يحدد ما إذا كان حساساً
  5. النظام يرسل للمدير طلب مراجعة
  6. المدير يراجع ويوافق/يعدّل
  7. النظام يحدّث Institution Profile + Views تلقائياً
- **التدفقات البديلة**: إذا كان الجدول الجديد حساساً، يحظره النظام تلقائياً حتى مراجعة المدير.

## 9. الخلاصة

يقدم هذا المشروع حلاً برمجياً متكاملاً يجمع بين أحدث تقنيات الذكاء الاصطناعي التوليدي والمعمارية المؤسسية الحديثة، مع التركيز على:

| الميزة | الوصف |
|--------|-------|
| 🎯 **اللغة العربية** | كأولوية رئيسية |
| 🔒 **الأمان والخصوصية** | كأساس لا يقبل المساومة |
| 🤖 **معمارية الوكلاء المتعددين** | كابتكار علمي (LangGraph) |
| 🗄️ **طبقة المشاهد المبسطة** | كحل هندسي ذكي لرفع الدقة |
| ⚡ **Smart Onboarding** | كتميز تنافسي في التخصيص |
| 🧠 **نموذج لغوي واحد قابل للتبديل** | كبساطة معمارية ومرونة |
| 🏗️ **القابلية للتخصيص** | لأي مؤسسة تعليمية أو إدارية |

المشروع يواكب أحدث التوجهات في علم الـ Agentic AI، حيث يعتمد على **LangGraph** لتنسيق الوكلاء المتخصصين (Supervisor, SQL, RAG, Analyzer)، بدلاً من التوجيه البسيط التقليدي. كما يحل مشكلة دقة Text-to-SQL عبر **طبقة المشاهد المبسطة** التي تُنشأ تلقائياً أثناء Smart Onboarding. ويعتمد على **نموذج لغوي واحد قابل للتبديل** (محلي عبر Ollama أو سحابي عبر API) يُدار بالكامل من لوحة التحكم، مما يبسّط البنية التحتية ويتيح لكل مؤسسة اختيار ما يناسبها.

الالتزام بالجدول الزمني المحدد (8 أشهر) والمعايير الأكاديمية يضمن إنتاج مشروع تخرج عالي الجودة يثبت إتقان الطلاب لمهارات هندسة البرمجيات والذكاء الاصطناعي. كما أن الباب مفتوح للتوسعة المستقبلية لتشمل Multi-Tenant وتطبيقات المؤسسات المختلفة.
